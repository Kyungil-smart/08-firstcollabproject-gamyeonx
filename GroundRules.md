# Commit Message

- [Init] 프로젝트 초기 세팅(첫 셋업, 폴더, 씬, 패키지 등)
    - 이후 작업 들어가면 사용X

- [Add] 기능/파일 추가(새 기능, 새 스크립트, 새 UI) : 없던 기능이 생김

- [Update] 기능 개선/변경(기존 기능 동작 변경, 밸런스 조정) : 있던 기능의 동작이 바뀜

- [Fix] 버그 수정(동작 오류, 크래시, 예외) : 동작이 안되던걸 코드 정리만

- [Remove] 삭제(기능/파일 제거)

- [Rename] 이름 변경(클래스/파일/폴더명)

- [Reractor] 리팩터(동작 동일, 구조만 개선) : 동작은 그대로, 코드 정리만 (함수 분리, 중복 제거)

- [Docs] 문서/README/주석 정리

기능 단위로 할 수 있도록 노력하기

# Folder

- 각자 개인적으로 구현하는 기능(스크립트, 프리펩 등)은 개인 이니셜(폴더)에서 작업
  - ex) 송근형 = SGH(폴더)에서 작업
    - 개인 폴더안 정리는 각자 알아서
  
  - 각자 작업이 끝났다고 판단되면, 공용 폴더에 작업물 이전
    - ex) 플레이어 스크립트의 경우 = Scripts(폴더) -> Player(폴더)로 옮김
  
  - 이후 버그 수정등은 이전 한 폴더에서 진행
    - ex)Scripts(폴더) -> Player(폴더)에서 진행
    - 버그 픽스 작업전에만 전파 해주면 됩니다.
  
  - 에셋은 Import(폴더)에서 관리
    - 에셋 배포는 팀장이 패키지화 후 배포
  
  - 폴더는 큰 카테고리 폴더 -> 세부 카테고리 폴더로 정리
    - ex) Prefabs(폴더) -> Monsters(폴더)

# Code

- 인터페이스 = 앞에 `I`를 붙인다.
  - ex)`IDamageable`

- 열거형 = 앞에 `E`를 붙인다.
  - ex)`EItem`

- public = 대문자 부터 시작
  - ex)`Attack`

- private = _소문자 부터 시작
  - ex) `_attack`

- Property(프로퍼티) = 소문자 부터 시작
  - ex) `attack`

- 함수 = 대문자 부터 시작
  - ex) `PlayerAttack()`

- 한 줄 단위의 코드는 상황에 따라 `{}` 생략 혹은 `=>`(람다식) 사용 권장
  - ex) `if (Count == 0) retun fales;`
  - ex) `get => _attack`, `set => _attack = value;` 등

- 들여 쓰기는 탭 한번 권장<br>
ex)
```cs
if (Count == 0)
{
//탭 한번        
}
```

- 변수명은 최대한 풀어서 선언 권장
  - ex) PlayerATK(X), PlayerAttack(O)
  - ex) HP, MP만 제외

# 파일명

- 앞에 Player, Monster, Ui 등 큰 카테고리 뒤에 기능등을 기입 권장
  - ex) PlayerMove, MonsterState, UiScore 등
  - ex) OrcState 등도 가능, 상황에 따라서는 융통성있게

- Prefab(프리팹)
  - 혹여나 기존 프리펩을 유지해야 하는 경우
    - ex) Player, Player02, Player03 이런 식으로 작성
    - 뒷 번호가 클 수록 최신
    - 만약 앞의 버전의 프리팹이 지워져도 네이밍 유지하길 권장
